## 3. 학습 정리

### 1. OHDSI 커뮤니티와 OMOP-CDM이 무엇이고 왜 필요한지
OHDSI는 관찰 의료 데이터를 활용하여 신뢰할 수 있는 근거를 만들기 위한 국제 연구 커뮤니티입니다.

데이터로 근거를 만든다는 것은  
실제 환자 기록을 분석해서 치료, 정책, 연구 판단에 쓸 수 있는 수치화 된 판단 근거를 만드는 것입니다.

OMOP-CDM은 병원마다 서로 다른 구조로 저장된 의료 데이터를 공통된 테이블 구조로 변환하기 위한 공통 데이터 모델입니다.

이때 중요한 것은,  
OMOP-CDM를 활용하는 분석 방식은 환자 데이터를 연구자 컴퓨터로 가져와서 분석을 하는 것이 아니라  
연구자가 만든 분석 코드를 병원 데이터베이스쪽으로 보내서 실행하는 방식이라는 것입니다.

데이터를 한 곳으로 모으지 않아도, 각 기관이 같은 구조의 데이터를 갖고 있기에 동일한 분석을 각자 실행할 수 있는 구조인 것입니다.

데이터를 연구자에게 가져오는 것이 아니라, 분석을 데이터가 있는 곳으로 보내는 방식이 OHDSI의 중요한 철학이라고 이해했습니다.

### 2. CDM 테이블 구조
OMOP-CDM은 의료 데이터를 공통 구조로 저장하기 위해서 여러 표준 테이블을 제공합니다.
PERSON, OBSERVATION_PERIOD, VISIT_OCCURRENCE, CONDITION_OCCURRENCE, DRUG_EXPOSURE, MEASUREMENT, PROCEDURE_OCCURRENCE, OBSERVATION, LOCATION, CARE_SITE, PROVIDER 등이 있습니다.

각 테이블에 대해서 간단히 설명을 적어두겠습니다.  
**PERSON**  
환자 기본 정보를 저장하는 테이블입니다.  
성별, 출생연도, 인종, 민족 등의 정보가 들어가며, 대부분의 임상 사건 테이블들은 PERSON을 기준으로 연결됩니다.  
그래서 PERSON은 분석의 기준이 되는 마스터 테이블처럼 이해했습니다.

**OBSERVATION_PERIOD**
해당 환자를 데이터 안에서 신뢰하고 관찰할 수 있는 기간을 담는 테이블입니다.  
데이터베이스가 어떤 사람의 의료 사건을 관찰할 수 있다고 판단하는 기간입니다.  
청구 기록 데이터의 경우에는 환자 개인의 보험 가입, 탈퇴, 재가입에 따라 한 사람에게 여러 개의 Observation Period가 생길 수 있습니다.  
반면에, 전자의무기록 데이터의 경우에는 병원 등록의 시작과 끝을 하나의 관찰 기간으로 봅니다.  

**VISIT_OCCURRENCE**  
환자의 병원 방문 정보를 저장하는 테이블입니다. 외래, 입원, 응급 방문과 같은 방문 단위의 기록이 들어갑니다.  
진단, 약물, 검사, 시술 같은 임상 사건들이 연관되는 중심 테이블이 됩니다.

**CONDITION_OCCURRENCE**  
진단이나 질병 정보를 저장하는 테이블입니다.  
예를 들어 제2형 당뇨병, 고혈압, 심부전 같은 진단 기록이 해당 테이블에 들어가게 됩니다.

**DRUG_EXPOSURE**  
약물 처방이나 투약 정보를 저장하는 테이블입니다.  
예를 들어 Metformin 처방, 인슐린 사용, 항생제 투약 기록 등이 해당 테이블에 들어가게 됩니다.

**MEASUREMENT**  
검사 결과나 측정값을 저장하는 테이블입니다.  
예를 들어 HbA1c, 혈압, 혈당, 콜레스테롤 검사 결과 등이 해당 테이블에 들어가게 됩니다.

**PROCEDURE_OCCURRENCE**  
시술이나 처치 정보를 저장하는 테이블입니다.

**OBSERVATION**  
애매하지만 연구에 의미 있는 관찰 정보를 저장하는 테이블입니다.
진단, 약물, 검사, 시술로 명확하게 분류하기 어려운 관찰성 정보를 저장합니다.  
예를 들어 흡연 상태, 가족력, 생활습관 정보 등이 들어갈 수 있습니다.

**LOCATION**  
지역, 주소 정보 테이블입니다.

**CARE_SITE**  
진료가 이루어진 병원, 부서, 진료 장소의 정보를 담는 테이블입니다.

**PROVIDER**  
의료진(진료 제공자) 정보를 담는 테이블입니다.  

LOCATION, CARE_SITE, PROVIDER 테이블은 환자에게 발생한 임상 사건 그 자체라기보다는, 사건이 어디에서, 누구에 의해 발생했는지를 설명하는 참조 테이블에 가깝다고 이해했습니다.

---

CDM 스키마와 Result 스키마에 대한 차이를 정리하자면,
CDM 스키마는 PERSON, CONDITION_OCCURRENCE, DRUG_EXPOSURE 같은 보호해야하는 표준 환자 데이터가 저장되는 공간이고, Result 스키마는 분석 도구가 생성한 코호트 결과, 분석 중간 결과, 통계 결과 등이 저장되는 공간이라고 이해했습니다.  
CDM 스키마 같은 경우는 보호하는 것이 매우 중요하기에 읽기 전용으로 보호하고, Result 스키마 같은 경우는 분석 도구가 자유롭게 결과를 쓰는 작업 공간 역할이기때문에 읽기와 쓰기를 모두 가능하게 하는 것이 일반적입니다.

### 3. 표준 용어 체계
OMOP-CDM에서는 의료 데이터를 단순히 원천 코드 그대로 분석하지 않고, OMOP Vocabulary의 Concept를 사용해 표준화합니다.  
Concept는 OMOP Vocabulary 안에서 하나의 의료 의미를 나타내는 표준 단위입니다.  
예를 들어 제2형 당뇨병, Metformin, HbA1c 검사, 남성, 외래 방문 등이 각각 하나의 Concept가 될 수 있습니다.  
각 concept는 concept 테이블에 하나의 Row들로 저장됩니다.  
concept_id, concept_name, vocabulary_id, domain_id, standard_concept 같은 정보가 포함됩니다.  

vocabulary_id는 이 concept가 어떤 코드체계나 용어집에서 온 것인지를 나타냅니다.  

domain_id는 이 concept가 어떤 종류의 의료 정보인지를 나타냅니다. 예를 들어 Condition, Drug, Measurement, Procedure, Visit 같은 도메인이 있습니다. 이 도메인은 해당 concept가 CDM의 어떤 테이블로 가야 하는지 판단하는 데에 중요한 기준이 되는 정보입니다.  

예를 들어 원천 데이터에 KCD-7 진단 코드가 들어있더라도, 그 코드가 매핑된 standard concept의 domain이 Procedure라면 CONDITION_OCCURRENCE가 아니라 PROCEDURE_OCCURRENCE에 기록되어야 할 수 있습니다.  

---

Source Value, Source Concept, Standard Concept의 차이는,  
**Source Value**는 원천 데이터에 실제로 기록된 값, 즉 병원 DB에 실제로 적혀있던 글자나 코드입니다.  
**Source Concept**는 원천 코드체계의 concept입니다.  
원천 코드를 OMOP Vocabulary 안에서 표현한 concept로, 원천 코드가 OMOP Vocabulary 안에서 어떤 concept로 표현되는지를 의미합니다.  
**Standard Concept**는 OMOP에서 분석에 사용하기 위해 정한 표준 concept입니다.  
여러 기관에서 서로 다른 코드체계를 사용하더라도, 같은 의미의 의료 개념을 하나의 표준 concept로 맞추어 분석할 수 있게 합니다.  

예를 들어 어떤 병원에서는 제2형 당뇨병을 KCD 코드로 저장하고, 다른 기관에서는 ICD 코드로 저장할 수 있습니다. 하지만 OMOP에서는 이 코드들을 표준 concept로 매핑하여 같은 의미로 분석할 수 있게 합니다.  

Concept들 사이에는 relationship이 존재하는데,  
relationship이 존재하는 이유는
1. 원천 코드를 표준 concept로 바꾸기 위해
2. 상위 개념으로 하위 개념을 한번에 찾기 위해
3. 약물, 검사, 시술 같은 복잡한 의미 구조를 표현하기 위해  

라고 정리했습니다.  

표준 용어 체계와 관련된 도구로는 Athena, PALLAS, Usagi가 무엇인지 학습했습니다.  

Athena는 OMOP Vocabulary를 검색하고 다운로드할 수 있는 웹 도구입니다.  
사용자는 Athena에서 concept를 검색하거나, 필요한 vocabulary를 다운로드할 수 있습니다.  

PALLAS는 여러 외부 용어 체계를 OMOP Standardized Vocabulary로 통합, 갱신하는 빌드 시스템입니다.  
일반 사용자가 직접 다루는 도구라기보다는, OMOP Vocabulary를 생성하는 내부 시스템으로 이해했습니다.  

Usagi는 원천 코드와 OMOP Standard Concept간의 매핑을 보조하는 도구입니다.  
병원 자체 코드나 원천 코드명을 입력하면 OMOP Vocabulary 안에서 비슷한 표준 concept 후보를 추천해주고, 사람이 최종 검토하여 매핑을 확정하는 방식입니다.  

**PALLAS가 만들고, Athena가 보여주고, Concept가 분석에 쓰인다.**
라고 이해했습니다.

### 4. ETL 과정에 사용되는 도구, 절차와 방법, 규칙
ETL은 원천 의료 데이터를 OMOP-CDM 구조로 변환하는 과정입니다.  

ETL 과정에 사용되는 대표적인 도구는 White Rabbit, Rabbit-in-a-Hat, Usagi가 있습니다.  

**White Rabbit**은 원천 데이터를 프로파일링하는 도구입니다.
ETL 설계 전에 원천 데이터에 어떤 테이블과 컬럼이 있는지, row 수는 얼마나 되는지, 값의 분포는 어떤지, 결측은 얼마나 있는지 등을 스캔하여 Scan Report를 생성합니다.  
원천 의료 데이터가 어떻게 생겼는지를 파악하고, 이후 ETL 설계의 근거를 만드는 도구로 이해했습니다.  
-> 원천 데이터 파악

**Rabbit-in-Hat**을 처음에 학습할때는 DataStage 같은 ETL 실행 도구는 아닙니다.  
실제 데이터를 변환하고 실행하는 도구라기보다는, 원천과 CDM 사이의 매핑 설계서를 GUI로 만들고 문서화하는 도구에 가깝다고 이해했습니다.  
-> CDM 변환 설계  

**Usagi**는 원천 코드와 OMOP Standard Concept를 매핑하는 데 도움을 주는 도구입니다. 병원 자체 코드나 원천 코드명을 OMOP Vocabulary의 표준 Concept 후보와 연결해주고, 사람이 최종 검토하여 매핑을 확정하는 방식입니다.  

OMOP Vocabulary에 이미 존재하는 원천 코드는 기존 매핑을 사용할 수 있지만, 병원 자체 코드처럼 OMOP Vocabulary에 없거나 매핑이 부족한 코드는 별도의 매핑 작업이 필요합니다. 이때 모든 코드를 사람이 수작업으로 매핑하기는 어렵기 때문에, 자주 쓰이는 코드부터 우선 매핑하거나, 기존 코드 관계를 활용해서 매핑하거나, Usagi 같은 매핑 보조 도구를 사용할 수 있습니다.

---

OMOP-CDM ETL에서는 어떤 테이블부터 채워야 하는지도 중요합니다.  
일반적으로,  
환자 기본 정보 -> 관찰 가능 기간 -> 병원/의료진 정보 -> 방문 정보 -> 진단, 처방, 검사 같은 실제 임상 사건  
순서로 테이블을 채웁니다.  

---  

CDM 테이블 구조만을 보고서는 ETL 과정에서의 모든 판단을 결정할 수 없습니다.  
예를 들어 날짜가 없는 경우 어떻게 처리할지, 매핑되지 않은 concept_id를 어떻게 처리할지 등이 있습니다.  
이러한 상황에서 판단을 기관마다 다르게 처리하면 같은 원천이 의미가 서로 다른 CDM 형태로 변환될 수 있기 때문에, 공통된 ETL convention이 필요합니다.  
THEMIS는 이러한 OMOP-CDM ETL convention을 논의하고 정리하는 OHDSI 커뮤니티 프로젝트입니다.

### 5. 데이터 품질 검증
DQ(= Data Quality)는 OMOP-CDM 데이터가 연구에 사용할 수 있을 만큼 신뢰할 수 있는지 확인하는 과정입니다.  
데이터 품질 검증 과정에서 가장 중요한 질문은
```
연구 목적에 적합한 데이터인가?
```
입니다.  

데이터 품질 문제는 수집, 저장, 변환, 분석, 배포 전 과정에서 발생할 수 있습니다.  
원천 데이터의 문제는 고칠 수 없을 수도 있지만, CDM 변환 과정에서 생기는 문제는 통제와 검증이 가능합니다. 따라서 CDM으로 변환되었다고 끝나는 것이 아니라, 이 데이터가 일반적으로도 괜찮고, 하려는 연구에도 적합한지를 계속 검증하는 것이 필요합니다.
